# Masri vs. MSA: When to Use Each Dialect

## What Are They?
- MSA (Modern Standard Arabic): Formal, written, pan-Arab
- Masri (Egyptian Arabic): Informal, spoken, Cairo-based

## Key Differences
| Aspect | MSA | Masri |
|--------|-----|-------|
| Register | Formal | Casual |
| Medium | Written | Spoken/digital |
| Audience | Scholars, formal | General population |
| Familiarity | Learned | Native (40M speakers) |

## When to Use MSA
- Official documents (contracts, regulations)
- Scholarly writing (research, books)
- News broadcasts (formal journalism)
- Religious content (Quran, formal prayer)
- Formal announcements

## When to Use Masri
- Social media (Instagram, TikTok, Twitter)
- Advertising (especially consumer brands)
- Informal blogs (personal voice)
- Marketing copy (relatable tone)
- Casual communication (chatbots, customer service)

## Real Examples
- E-commerce site: Masri (friendly, relatable)
- Government portal: MSA (official, formal)
- Meta ad for fitness app: Masri (casual, energetic)
- Academic paper: MSA (formal, authoritative)

## The Trap: Mixing Them
- Why mixing confuses readers
- How AI often mixes (defaulting to formal MSA)
- Cost of mistakes (brand damage, lost sales)

## Practical Advice
- When unsure, ask: "Would a native say this over coffee or in a newspaper?"
- Coffee → Masri | Newspaper → MSA
- Masri is growing in digital; MSA in formal contexts

## Tools & Resources
- Awesome Arabic Skill: 11 dialects including Masri
- Dialect selection guide (link)
- Example content library (link)

## Conclusion
- Dialect choice = audience trust
- Know your audience, choose your dialect
- Call-to-action: "Try dialect routing in Awesome Arabic Skill"

# RTL & Bidirectional Text: A Developer's Guide

## Why RTL Matters
- Right-to-Left languages: Arabic, Hebrew, Persian, Urdu
- ~300M native speakers
- Common bugs: broken layouts, reversed punctuation, unreadable mixed text

## The Problem
- CSS direction: easy (`dir="rtl"`)
- Unicode bidirectional text: hard (RLE, PDF markers)
- Mixed Arabic+English: chaos without proper marks

## Unicode Markers
- RLE (U+202A): Right-to-Left Embedding
- LRE (U+202D): Left-to-Right Embedding
- PDF (U+202C): Pop Directional Formatting
- LRM (U+200F): Left-to-Right Mark
- RLM (U+200E): Right-to-Left Mark

## Common Mistakes
1. Forgetting LRM after Arabic→English transition
   - ❌ "تطبيق Fitness" (looks: "Fitness تطبيق")
   - ✅ "تطبيق‏ Fitness" (with LRM after "تطبيق")

2. Unbalanced RLE/PDF
   - ❌ "RLE Arabic RLE more Arabic" (extra RLE)
   - ✅ "RLE Arabic PDF RLE more Arabic PDF" (balanced)

3. Not setting dir="rtl" on HTML
   - Always set on root or parent container

## Validation Checklist
- [ ] HTML dir="rtl" attribute
- [ ] RLE/PDF markers balanced
- [ ] LRM present after Arabic→English transitions
- [ ] Font supports Arabic (e.g., IBM Plex Sans Arabic)
- [ ] Line breaks don't reverse punctuation

## Tools
- Awesome Arabic Skill: validate-rtl.sh (automated checks)
- Browser DevTools: inspect bidi text
- Unicode analyzer: check character codes

## Code Example
```typescript
// ❌ Wrong
<p>Hello تحيا مصر Welcome</p>

// ✅ Correct
<p dir="rtl">
  <span>Hello‏</span> تحيا مصر <span>Welcome</span>
</p>
```

## Performance Considerations
- RTL re-layout can be expensive (but negligible on modern browsers)
- Cache validated text (don't re-validate on every render)
- Use CSS containment if you have many RTL sections

## Resources
- Awesome Arabic Skill: RTL audit guide
- MDN: Structural markup and right-to-left text
- W3C: Structural Markup and Right-to-Left Text

## Conclusion
- RTL is not an edge case (300M speakers)
- Validation is essential (Awesome Arabic Skill automated)
- Small effort, big impact (trust from Arab users)

## Call-to-Action
- "Use Awesome Arabic Skill's RTL validator in your CI pipeline"
- Link to validate-rtl.sh

# Why Load Presets Matter: Task-Class Bundling for Efficiency

## The Problem
- Awesome Arabic Skill has 26 reference files (audit guides, dialects, domains, etc.)
- Loading all 26 files for every request = wasted tokens + slow responses
- But different tasks need different files
  - Writing a caption: need tone, dialect, humanization
  - Auditing content: need audit-mode, RTL, dialect-purity
  - Planning campaign: need seasonal-calendar, market-strategy

## The Solution: Load Presets

A **load preset** is a curated bundle of files for a specific task:

```
Preset "seasonal" = [
  seasonal-calendar.md
  seo-aeo-masri.md
  humanization-protocol.md
]

Preset "audit" = [
  audit-mode.md
  validate-rtl.sh
  validate-dialect-bleed.sh
]
```

## Available Presets
1. **plan** — Planning content (SEO, campaign, book)
2. **write** — Drafting content (captions, ads, copy)
3. **audit** — QA & review
4. **seasonal** — Holiday-specific (Ramadan, Eid, New Year)
5. **campaign** — Marketing campaigns (ads, landing pages)
6. **book** — Long-form (chapters, book content)
7. **seo-aeo-masri** — Egyptian SEO/AEO
8. **seo-aeo-gulf** — Gulf SEO/AEO
9. **seo-aeo-ksa** — Saudi SEO/AEO
10. **seo-aeo-levantine** — Levantine SEO/AEO
11. **coach** — Prompt improvement
12. **audit-full** — Complete audit (all reference files)
13. **dialect-lock** — Force one dialect (no bleed)
14. **init** — Onboarding (first-time setup)

## Benefits

### 1. Speed
- Smaller context window
- Faster LLM processing
- Example: write preset loads 3 files vs. 26 → 80% context savings

### 2. Cost
- Fewer tokens used per request
- Lower API costs
- Example: 1000 captions × 20% fewer tokens = significant savings

### 3. Accuracy
- Only relevant files loaded
- Less "noise" to confuse the model
- Focus on what matters for the task

### 4. Consistency
- Same files every time
- Reproducible results
- No "version creep" from different file selections

## How to Use

### Via CLI
```bash
/arabic write caption --preset seasonal
/arabic audit --preset audit-full
bash scripts/load-preset.sh plan
```

### Via Plugin API
```json
{
  "type": "caption",
  "dialect": "masri",
  "brief": "Ramadan greeting",
  "preset": "seasonal"
}
```
