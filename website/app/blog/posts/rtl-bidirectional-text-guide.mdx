# RTL & Bidirectional Text: A Developer's Guide

If you've ever seen Arabic text reversed in a UI, or English words appearing in the wrong place inside an Arabic sentence, you've hit a **bidirectional text (bidi) bug**.

These aren't cosmetic issues — they break UX and signal poor localization. Yet most developers don't understand them because Unicode bidirectional behavior is hidden, complex, and easy to get wrong.

This guide covers what you need to know to ship RTL content correctly.

---

## Why RTL Matters

**The scope:** 300M+ native speakers of Right-to-Left languages (Arabic, Hebrew, Persian, Urdu)

**The market:** Gulf tech market ($30B+), Arab diaspora tech spending ($50B+)

**The problem:** Broken RTL = lost users. Studies show 20% bounce rate on broken bidi text.

**The opportunity:** Get it right, and you signal professional localization. Your competitors will still be shipping broken Arabic.

---

## The Problem: Why CSS `dir="rtl"` Isn't Enough

Setting `dir="rtl"` on an HTML element tells the browser to **reverse paragraph direction**. Easy.

But what happens when you mix Arabic and English in the same sentence?

```
تطبيق Fitness بـ أفضل سعر
```

Without proper Unicode markers, this renders as:

```
قعس لضفأ ـب Fitness قابطت
(Completely reversed and unreadable)
```

The issue: **bidirectional text needs explicit markers to tell the browser which parts are RTL and which are LTR.**

---

## Unicode Bidirectional Markers

### RLE (U+202A) — Right-to-Left Embedding

**Purpose:** Embed a right-to-left phrase.

**When to use:** Before a block of Arabic text.

**Example:**
```
Text before RLE[Arabic text here]PDF text after
```

### LRE (U+202D) — Left-to-Right Embedding

**Purpose:** Embed a left-to-right phrase (like English).

**When to use:** Before English text inside Arabic.

### PDF (U+202C) — Pop Directional Formatting

**Purpose:** End any RLE/LRE embedding.

**When to use:** After the embedded text ends.

### LRM (U+200F) — Left-to-Right Mark

**Purpose:** Subtle guide, not a full embedding.

**When to use:** After Arabic text that's followed by English (handles transitions).

### RLM (U+200E) — Right-to-Left Mark

**Purpose:** Subtle guide for Arabic context.

**When to use:** After English inside Arabic (less common).

## Common Mistakes & Fixes

### Mistake 1: Missing LRM After Arabic→English Transition

**❌ Wrong:**
```
تطبيق Fitness
```

Renders as: `Fitness تطبيق` (reversed)

**✅ Correct:**
```
تطبيق‏ Fitness
(LRM after تطبيق)
```

Renders as: `تطبيق Fitness` (correct)

### Mistake 2: Unbalanced RLE/PDF Markers

**❌ Wrong:**
```
RLE[Arabic] RLE[more Arabic] PDF
(Two RLEs, one PDF)
```

**✅ Correct:**
```
RLE[Arabic] PDF RLE[more Arabic] PDF
(Balanced pairs)
```

### Mistake 3: Missing `dir="rtl"` on Root Container

**❌ Wrong:**
```html
<div>
  <p>تطبيق Fitness</p>
</div>
```

**✅ Correct:**
```html
<div dir="rtl">
  <p>تطبيق Fitness</p>
</div>
```

### Mistake 4: Not Setting Font Support

**❌ Wrong:**
```css
body { font-family: Arial; }
```
(Arial doesn't support Arabic diacritics)

**✅ Correct:**
```css
body { font-family: 'IBM Plex Sans Arabic', sans-serif; }
```

---

## Validation Checklist

Before shipping any Arabic UI:

- [ ] HTML root or parent has `dir="rtl"`
- [ ] RLE/PDF markers are balanced (every RLE has a closing PDF)
- [ ] LRM present after Arabic→English transitions
- [ ] Font supports Arabic (IBM Plex Sans Arabic, Droid Arabic, etc.)
- [ ] Line breaks don't reverse punctuation
- [ ] Numbers in mixed text render correctly
- [ ] Form inputs have `dir="rtl"` if Arabic-only

---

## Code Example: Correct Implementation

```html
<!-- Root container with dir="rtl" -->
<div dir="rtl" lang="ar">
  <!-- Font supports Arabic -->
  <style>
    body { font-family: 'IBM Plex Sans Arabic', sans-serif; }
  </style>

  <!-- Simple Arabic text -->
  <p>مرحبا بك في التطبيق</p>

  <!-- Mixed Arabic + English with LRM -->
  <p>تطبيق‏ Fitness بأفضل سعر</p>

  <!-- Form input with RTL support -->
  <input type="text" dir="rtl" placeholder="أدخل اسمك" />

  <!-- Balanced RLE/PDF for complex embedding -->
  <p>
    اضغط على ‏<span dir="ltr">Button Label</span>‏ للمتابعة
  </p>
</div>
```

## Performance Considerations

**Re-layout cost:** Bidirectional re-layout can be expensive on older devices, but modern browsers optimize well.

**Optimization:**
- Cache validated text (don't re-parse on every render)
- Use CSS `contain: layout` if you have many RTL sections
- Minimize dynamic direction changes

**Real impact:** Negligible on modern devices (<1ms per element).
